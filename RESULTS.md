# Benchmark Results

Evaluated on `samples.json` — 100 synthetic OCR-style documents generated by `generate_samples.py`
(`--num-samples 100 --seed 42`).  
Hardware: CPU only (Intel, laptop-class).  
Model: `openai/privacy-filter` loaded in bfloat16.

### False positive breakdown

375 FPs total, of which **199 (53%) hit known-benign values** — things the model should not flag:

| Benign category | FP count | Share of all FPs |
|---|---|---|
| Admin dates (`private_date`) | 167 | 44.5% |
| Business addresses | 29 | 7.7% |
| Company names (person-like) | 3 | 0.8% |
| Other / unexplained | 176 | 46.9% |

The model aggressively tags all dates as `private_date`, regardless of context. Invoice dates, pay periods, statement dates, and transaction dates all trigger it. This is the single largest source of FPs and the clearest area for improvement (see [Observations](#observations)).

## Address masking

| Metric | Value |
|---|---|
| Matched addresses | 71 |
| Full address masked (≥ 85% coverage) | 47 (66%) |
| Partial mask (< 85% coverage) | 24 (34%) |
| Avg coverage ratio | 0.76 |

Coverage ratio = `len(predicted span) / len(ground-truth address string)`. 1.0 means the model captured the full address as a single span; lower means it captured only part.

Partial match examples:

| Ground truth | Model prediction | Coverage |
|---|---|---|
| `250 Ramos Street, West Adrian, IN 94935` | `250 Ramos Street, West Adrian` | 0.74 |
| `Nergerstr. 573, 08734 Ueckermünde` | `08734` | 0.15 |
| `北京市荆门县梁平乌鲁木齐街T座 541070` | `541070` | 0.27 |

The German and Chinese address failures are telling: the model extracted only the postal code, missing the street and city entirely. Non-English address formats (abbreviated street types like `Nergerstr.`, CJK characters) are a clear weakness.

## Observations

### 1. Date over-tagging is the dominant problem
167 of 375 FPs (44.5%) are administrative dates tagged as `private_date`. The model does not distinguish between a date of birth and an invoice date. Post-processing rules or a date-context classifier on top of the model's output would substantially improve precision.

### 2. W-2 and 1099 accuracy is limited by masked SSNs
The templates show SSNs as `XXX-XX-1234` (as they appear in real documents). The model never sees the full SSN, so it cannot tag it, creating unavoidable false negatives. Recall for these types would be higher on documents that contain unmasked SSNs.

### 3. Non-English address recall drops sharply
German abbreviated street names and CJK-script addresses are partially missed. The model captures postal codes reliably but struggles with non-English street and city components.

### 4. Person-name company false positives are minimal (3 total)
The ~60% person-name company name generation strategy produced very few false positives, suggesting the model handles business-context names reasonably well. This was one of the hypothesised weak spots and is not a significant issue in practice.

### 5. Throughput is viable for batch processing
363 tokens/sec on CPU means a typical 400-token document processes in ~1 second. For offline batch redaction workloads this is acceptable; for real-time interactive use a GPU or the model's WebAssembly deployment would be preferred.

## Reproducing

```bash
# Generate 100-sample dataset
python generate_samples.py --num-samples 100 --seed 42 --output samples.json

# Run benchmark (suppress per-sample output)
python infer.py --input samples.json --benchmark --quiet
```
